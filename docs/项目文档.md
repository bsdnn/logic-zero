## 1. 项目背景

### 1.1 任务定义

**Knights & Knaves**（骑士与流氓）是一类经典逻辑谜题：

> 岛上有两种人：骑士 (Knight) 永远说真话，流氓 (Knave) 永远说假话。
>
> A 说："B 是流氓。"
> B 说："A 和我都是骑士。"
>
> 请判断 A 和 B 各是什么身份。

每道题给出 n 个人（n=2~7）和他们的陈述，要求模型推理出每个人的身份。这是个**有唯一标准答案**的逻辑任务，且**可程序化验证**——非常适合作为评估"推理能力"的小型 benchmark。

### 1.2 项目目标

- 用一个 1.5B 的开源小模型（Qwen2.5-1.5B），跑完业界标准的**三阶段后训练流水线**：SFT → DPO → GRPO
- 在不依赖云端大算力的前提下，把准确率从 0% (base model) 提升到尽可能高
- 全程在免费/低成本 Colab 上完成（A100 / L4 / T4）
- 沉淀一套可复现、可扩展到其他推理任务的代码框架

### 1.3 为什么是骑士与流氓？

| 优点 | 说明 |
|---|---|
| 答案可验证 | 用 SMT solver（Z3）能在毫秒级判定"这个答案是否唯一且正确" |
| 难度可控 | 通过调整人数 n，可以平滑地从简单（n=2，64 种状态）到复杂（n=7，128 种） |
| 不依赖世界知识 | 纯逻辑任务，不会因为"模型不知道某个事实"而失败 |
| 适合 reward 函数设计 | 答案正确/错误是清晰的二元信号，适合做 RL 训练 |

## 2. 基础知识

读项目代码之前，下面这些概念需要先建立。每个概念配一个"为什么需要它"的动机。

### 2.1 大语言模型训练的三个阶段

```
Pretraining → Post-training → Inference
   ↓               ↓                ↓
学语言        学怎么用语言        实际使用
(几T token)   (几B token)        (推理调用)
```

**Pretraining（预训练）**：在 TB 级别的互联网文本上做下一个 token 预测。学到的是"语言的统计规律"——给定上下文，下一个词最可能是什么。基模型（base model）就是这个阶段的产物。

**Post-training（后训练）**：在预训练好的基模型上，用相对少量但**质量更高的数据**，把模型调成一个"听话的助手"或"会推理的解题器"。这个项目做的就是后训练。

本项目用的 Qwen2.5-1.5B 是**已经做过基础 instruction-tuning 的版本**，但它在骑士流氓这种特定推理任务上准确率几乎是 0——后训练的目的就是把这部分能力做出来。

### 2.2 后训练的三个常见步骤：SFT → DPO → GRPO

#### SFT (Supervised Fine-Tuning，监督微调)

**做什么**：给模型看大量「输入 → 标准答案」的样本，让它学着模仿。

```
输入：A 说: B 是流氓。B 说: A 和我都是骑士。请判断身份。
输出：<think>分两种情况...假设 A 是骑士...</think>
      <answer>A: knight, B: knave</answer>
```

**目标函数**：让模型生成上述输出的对数似然最大化。本质上还是"下一个 token 预测"，只不过 token 来自精心准备的演示数据。

**类比**：像学生抄写老师的范文。学完之后能模仿出大致的格式和思路，但不一定真的"理解"了。

#### DPO (Direct Preference Optimization，直接偏好优化)

**做什么**：给模型看「同一个输入下的两个回答：A 比 B 好」的偏好对，让模型学会"产出 A 这种回答的概率更高，产出 B 这种回答的概率更低"。

**为什么 SFT 不够**：SFT 只告诉模型"什么是对的"，没告诉模型"什么是错的"。模型可能学会一种格式，但仍然产生大量错误推理。DPO 通过对比学习填补这一点。

**目标函数**（简化）：

```
L_DPO = -log σ( β · [ log π(chosen|x) - log π(rejected|x) - log π_ref(chosen|x) + log π_ref(rejected|x) ] )
```

直觉解释：
- `π` 是当前正在训练的模型
- `π_ref` 是冻结的参考模型（通常是 SFT 之后的 checkpoint）
- 鼓励"chosen 比 rejected 概率高"，但用 ref model 做锚定，防止模型跑得离原始能力太远
- `β`（beta）控制偏离参考模型的容忍度——越大越保守

**类比**：像批改作文。学生写了 A 和 B 两版，老师圈出 A 更好，学生下次倾向于写 A 的风格。

#### GRPO (Group Relative Policy Optimization)

**做什么**：DPO 是离线（offline）的——预先准备好偏好对就行了。GRPO 是**在线**（online）的——模型自己生成 K 个回答，根据每个回答的 reward 高低，把概率重新分配。

**为什么 DPO 不够**：DPO 用的是"人类/模型预先标好的偏好对"，覆盖范围有限。GRPO 让模型在训练中持续探索新的回答模式，按 reward 自适应调整。

**简化流程**（每一步）：

1. 模型对同一个 prompt 采样 K=4 个回答 (`rollouts`)
2. 对每个回答跑 reward function 得分（本项目：格式 0.5 分 + 答对 2 分 + 长度 0.3 分）
3. 把 K 个分数 z-score 标准化，得到每个回答的相对优势 (advantage)
4. 优势为正的回答 → 提升其概率；优势为负 → 降低
5. 同时加 KL 散度惩罚，防止偏离参考模型太远

**和 PPO 的区别**：经典 PPO 需要一个独立的 value head 估计基线 (baseline)。GRPO 直接用"同 prompt 下其他回答的平均分"作为基线，省掉一个网络——更省显存，更适合小模型/LoRA 场景。

**类比**：像考试。同一道题让学生做 4 遍（采样温度 > 0），按对错排序，把"做得对的那次"的解题策略加强，"做错的那次"的策略削弱。

### 2.3 LoRA (Low-Rank Adaptation)

**问题**：1.5B 模型有 15 亿参数。每个参数 fp16 占 2 字节，**单是权重就 3 GB**。训练时还要存梯度、优化器状态、激活值——一个 22GB 显存的 L4 直接爆炸。

**LoRA 的核心 idea**：训练的时候，**不动原始权重 W**，而是在它旁边加一个低秩分解 `ΔW = B·A`（两个小矩阵），只训练 A 和 B：

```
原始：  y = W·x        (W 是 d×d 大矩阵，比如 4096×4096)
LoRA：  y = W·x + B·A·x  (A 是 r×d，B 是 d×r，r 通常取 8/16/32)
```

如果 r=16，d=4096，A 和 B 加起来只有 `2 × 16 × 4096 = 131K` 参数，**比原来的 16M 少了 128 倍**。整个模型加起来 LoRA 适配器只占 17 MB，可以用消费级显卡训练。

**推理时**：可以选择把 LoRA 合并回原模型（`W' = W + B·A`），或者保持分离（每次 forward 多一次小矩阵乘法，开销可忽略）。

### 2.4 量化精度：bf16 / fp16 / fp32

- **fp32**：每个浮点数 32 bit。精度高，但显存占用大。
- **fp16**：16 bit，半精度。省一半显存但**指数位太少**（5 bit），容易溢出，训练 loss 容易变 NaN。
- **bf16** (brain-float-16)：也是 16 bit，但**指数位 8 bit、尾数位 7 bit**——和 fp32 同样的动态范围，精度差一点但稳定得多。**Ampere 架构以上的 GPU（A100、L4、RTX 30/40 系列）原生支持，是现在的默认选择。**

T4 (Turing 架构) 不原生支持 bf16，所以本项目代码会自动检测：

```python
use_bf16 = torch.cuda.is_available() and torch.cuda.is_bf16_supported()
```

### 2.5 SMT solver / Z3

**SMT** = Satisfiability Modulo Theories，一种自动定理证明器。Z3 是微软开源的 SMT 实现。

本项目用 Z3 干两件事：

1. **生成时验证唯一解**：每个骑士流氓谜题在生成时，要确认它的标准答案是**唯一的**（不能有多组身份组合都满足所有陈述），否则模型答对了也无法判断。
2. **训练评估时验证答案**：模型生成的 `<answer>A: knight, B: knave</answer>` 是不是和标准答案吻合。

代码大致长这样：

```python
import z3
s = z3.Solver()
A, B = z3.Bool('A'), z3.Bool('B')  # True = knight
# A 说 "B is knave" → A 的真值 == "B is knave" 的真值
s.add(A == z3.Not(B))
# B 说 "A and B are knights" → B 的真值 == (A ∧ B)
s.add(B == z3.And(A, B))
s.check()  # sat / unsat / unknown
model = s.model()  # {A: False, B: True} 之类
```

### 2.6 GPU 显存预算（直觉数）

| 项目 | 显存 |
|---|---|
| Qwen2.5-1.5B 权重（bf16） | ~3 GB |
| LoRA 适配器权重 + 梯度 + 优化器状态 | ~200 MB |
| 训练时的激活值（seq_len=1024, batch=2） | ~4-6 GB |
| DPO/GRPO 的 reference model（同样大小） | ~3 GB |
| GRPO 的 K 个 rollouts 同时驻留 | ~2-4 GB |
| **小计** | **~12-16 GB** |

L4 (22 GB) 刚好够，A100 (40/80 GB) 绰绰有余，T4 (16 GB) 训练吃力（DPO/GRPO 容易 OOM），但**推理评估完全够用**——这是本项目把训练放在 L4/A100、把评估放在免费 T4 的原因。

## 4. 数据流水线

### 4.1 谜题生成 `data/gen_puzzles.py`

每次生成需要：
- 随机选 n 个人（标签 A, B, C...）
- 给每个人随机分配身份（knight / knave）作为 ground truth
- 给每个人生成一条陈述（关于其他人身份、或者关于人数等）
- 用 Z3 验证：**仅给这些陈述、不告诉身份**，求解器能否得出唯一身份组合？
  - **唯一解** → 接受这道题
  - **多解或无解** → 重新生成

陈述类型（随机选）：
- `"X 是 knight/knave"`
- `"X 和 Y 身份相同/不同"`
- `"至少 K 个 knight/knave"`
- `"我是 knight/knave"`（自指）

### 4.2 SFT 数据合成 `data/gen_sft_data.py`

谜题本身没有思维链——只有题目和答案。SFT 需要演示**怎么推理出答案**的过程，所以我们让一个**更强的老师模型（DeepSeek-V3）**来写思维链：

```
输入给 DeepSeek-V3：
  谜题文本 + 标准答案 + 一个固定模板
  ("请用 <think>...</think> 包裹推理过程，<answer>...</answer> 给出答案")

输出：
  <think>
  让我们分析陈述...假设 A 是 knight，那么...这与...矛盾...因此 A 是 knave...
  </think>
  <answer>A: knave, B: knight</answer>
```

按 n 分桶配额（spec §5）：

| n | 数量 |
|---|---|
| 2 | 30   |
| 3 | 200  |
| 4 | 300  |
| 5 | 350  |
| 6 | 500  |
| **总计** | **~1380** |

n 大的题更难，所以分配更多样本帮助模型学习。

### 4.3 DPO 偏好对合成 `data/gen_dpo_data.py`

DPO 需要「同 prompt 下 chosen vs rejected」的对子。**让 SFT 模型自己生成**：

1. 对每道题用 SFT 模型采样 4 个回答（温度 T=0.8）
2. 用 Z3 验证每个回答的答案
3. 如果至少有一个**对的**和一个**错的**：
   - 取得分最高的对的回答 → `chosen`
   - 取得分最低的错的回答 → `rejected`
4. 如果全对或全错，跳过这道题

**为什么不用人类标注**：成本和扩展性。让模型自己标注（self-play）是 DeepSeek-R1、Llama-3 等大模型也在用的方法。

实际跑下来：因为时间预算（Colab 4 小时），最终收集到 **95 对**（计划 800+）。

### 4.4 GRPO 数据 `data/gen_grpo_data.py`

GRPO 不需要预先标注——只需要 prompt + ground truth（reward 函数会用）：

```json
{"prompt": "A 说: B 是流氓...", "ground_truth": {"A": "knight", "B": "knave"}, "n": 2}
```

按 n 分桶生成 1780 道（n=2 因为题空间限制只有 30 道，其余 250-500 道）。

## 5. 训练流水线

### 5.1 SFT 训练 `train/sft.py`

**关键配置：**

```python
config = SFTConfig(
    num_train_epochs=3,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,    # effective batch=16
    learning_rate=2e-4,                # LoRA 比全量微调可以用大 10x 的 LR
    bf16=True,                         # 自动检测，T4 上降到 fp16
    gradient_checkpointing=True,       # 用 20% 速度换 40% 显存
    max_seq_length=1024,
)
```

**Best checkpoint 选择**：自定义 `DevAccuracyCallback`，每个 epoch 结束在 dev 集上做 greedy generation，取准确率最高的那一版保存。原因：训练 loss 下降不等于准确率上升，必须以下游任务指标为准。

**输出**：17 MB 的 LoRA 适配器 `results/checkpoints/sft/best/adapter_model.safetensors`。

### 5.2 DPO 训练 `train/dpo.py`

**关键配置：**

```python
config = DPOConfig(
    num_train_epochs=2,
    learning_rate=5e-6,    # 比 SFT 小 40x —— DPO 容易把模型训崩
    beta=0.1,              # KL 系数，越大越保守
    max_length=1024,
    max_prompt_length=256,
)
```

需要**两份模型**同时驻留：
- `model` (trainable)：从 SFT checkpoint 继续训
- `ref_model` (frozen)：SFT checkpoint 的冻结副本

DPOTrainer 会自动比较两者的 log-probability，更新 `model` 朝偏好方向走。

### 5.3 GRPO 训练 `train/grpo.py`

**关键配置：**

```python
config = GRPOConfig(
    num_train_epochs=2,
    learning_rate=1e-6,            # 比 DPO 还小，RL 更不稳定
    beta=0.04,                     # KL 系数
    num_generations=4,             # 每个 prompt 采样几个 rollouts
    max_completion_length=512,
    temperature=1.0,               # 必须 > 0 才能采样
)
```

**Reward 函数** (`train/reward.py`):

```python
def compute_reward(response, ground_truth):
    score = 0.0
    if check_format(response):       # 有 <think>...</think><answer>...</answer>
        score += 0.5
    if extract_answer(response) == ground_truth:
        score += 2.0                 # 答对
    if len(response) > 80:
        score += 0.3                 # 防止模型偷懒只输出 <answer></answer>
    return score
```

设计原则：
- **格式分**给得低（0.5），避免模型只学格式不学推理
- **正确分**给得高（2.0），主导信号
- **长度分**作为防 hack 措施（reward hacking 经典 pattern：模型学会输出最短的合法格式）

## 6. 评估方法

### 6.1 评估集

- **测试集**：1530 道，覆盖 n=2~7（n=7 是分布外 OOD，训练时没见过）
- **生成方式**：与训练集同生成器但**不同随机种子**，且通过哈希去重确保没有重叠

### 6.2 指标

| 指标 | 含义 |
|---|---|
| **Overall accuracy** | 模型给出与 ground truth 完全一致的答案的比例 |
| **Per-n accuracy** | 按谜题人数 n 分桶的准确率 |
| **Format compliance** | 输出符合 `<think>...</think><answer>...</answer>` 格式的比例 |

### 6.3 解码策略

默认 **greedy decoding (T=0)**，原因：

- 这是个有标准答案的任务，确定性输出更可重现
- 比 3-seed sampled 快 4 倍
- 对小模型来说，sampling 引入的方差容易掩盖真实能力

`eval/run_eval.py` 也支持 `--with-sampled` 跑 3 个 seed 的 T=0.7 采样，但默认不开。

### 6.4 工程优化：断点续跑

1530 道题在 T4 上要 16 小时，Colab 90 分钟会断一次连接。所以评估器实现了：

- 每道题预测出来立刻写入 `eval_*.greedy.preds.jsonl`（流式 append）
- 每 25 道题镜像一次到 Google Drive
- 重连后启动时，从 Drive 拉回最新的 jsonl，跳过已完成的题，继续跑剩下的

代码里关键片段：

```python
if preds_path.exists():
    for line in open(preds_path):
        rec = json.loads(line)
        skip_set.add(rec["i"])    # 这道题已经做过了
```

## 7. 实验结果

### 7.1 完整数据（截至 GRPO 开训前）

| 指标 | Base (Qwen2.5-1.5B) | SFT | DPO |
|---|---|---|---|
| **Overall accuracy** | ~0% | 12.9% | **14.6%** |
| Format compliance | 极低 | 90.9% | 待测 |
| n=2 | - | - | 53.3% |
| n=3 | - | - | 27.3% |
| n=4 | - | - | 20.3% |
| n=5 | - | - | 11.3% |
| n=6 | - | - | 7.0% |
| n=7 (OOD) | - | - | 3.0% |

### 7.2 解读

**DPO 相对 SFT 提升 +1.7 pp（+13% 相对）**：在 1.5B 这个规模 + 只有 95 个偏好对的限制下，这是合理的范围。学术界对 1.5B 模型做 DPO 通常报 +1~3 pp。

**n=2 达到 53.3%**：说明经过两轮训练后简单逻辑题已经能稳定解出，模型学到了基本的"骑士说真话/流氓说假话"的演绎规则。

**难度曲线单调下降**：53 → 27 → 20 → 11 → 7%，没有崩盘式塌缩，说明模型不是完全靠记忆而是在尝试推理。

**n=7 仍有 3.0%**（OOD 泛化）：训练集只覆盖了 n=2~6，n=7 是没见过的难度。3% 看似低但**不为 0**，说明模型不是死记硬背特定模式，而是学到了某种可迁移的方法。

### 7.3 GRPO 预期

按文献和经验，GRPO 在小模型上能再带来 +3~10 pp 的提升，但**前提是 reward 函数设计合理 + 不出现 reward hacking**。本项目的 GRPO 训练在写完文档的时候还没启动，会用 stage A smoke test + mini-smoke 先验证可行性，再决定要不要全量跑。

## 8. 工程细节与踩坑

### 8.1 已经踩过的坑

| 现象 | 原因 | 解决 |
|---|---|---|
| 评估时模型在 CPU 上跑（极慢） | `load_base_model()` 返回 CPU 模型，没有显式 `.to('cuda')` | 加 `model = model.to('cuda')` + `assert next(model.parameters()).device.type == 'cuda'` |
| n≥5 的题准确率异常低 | `max_new_tokens=700` 不够，n=6 的推理 p95 ≈ 753 token，被截断了 | 提到 1000 |
| Eval 慢得离谱 | 模型生成到 max_new_tokens 才停 | 加 `stop_strings=["</answer>"]`，简单题快 10x |
| trl 0.13 报 `MODEL_FOR_VISION_2_SEQ_MAPPING_NAMES` ImportError | trl 版本与 transformers 不兼容 | 锁版本：`transformers>=4.46,<4.50` + `trl>=0.13.0,<0.15.0` |
| Colab 断了，几小时进度没了 | 没有持久化 | 实现 per-puzzle JSONL 流式 + 每 25 题 Drive 镜像 |
| DPO 数据合成跑了 4 小时只攒 95 对 | T=0.8 的接受率比预估低，加上 n=4/5/6 推理慢 | Option B 砍数据量；时间不够时直接停训 |
| 跑 DPO 训练突然报 `tokenizer` 参数错误 | trl 0.13 把 `tokenizer` 重命名为 `processing_class` | 用 try/except 双签名兼容 |

### 8.2 复现成本

| 阶段 | 平台 | 时间 | Colab 单元 |
|---|---|---|---|
| SFT 数据合成 | DeepSeek API | ~1h | $2-3 API 费 |
| SFT 训练 | L4 | ~2h | ~10 |
| SFT 评估 | T4 (免费) | ~16h | 0 |
| DPO 数据合成 | A100 | 4h（被截断） | ~50 |
| DPO 训练 | L4 | 86 秒 | <1 |
| DPO 评估 | T4 (免费) | ~16h | 0 |
| GRPO 训练 | L4 | ~6-8h（预计） | ~30-40 |
| GRPO 评估 | T4 (免费) | ~16h | 0 |

**总计**：约 100-130 Colab 单元 + $3 API 费用。整套流程在免费/Pro 账号上可行。

### 8.3 设计上的几个有意思的选择

**为什么用 LoRA 而不是全量微调？**
- 1.5B 模型全量微调 + Adam 优化器状态 = `1.5B × 4 字节 × 8(?) = ~12GB` 显存光给优化器。LoRA 把这部分降到几 MB。
- LoRA 的副作用是限制模型能学到的方向，但对**已经预训练好的模型做小幅度调整**的场景非常合适。

**为什么不直接用 PPO 而用 GRPO？**
- PPO 需要训练一个 value head 估计基线，对 1.5B 模型来说额外 ~150M 参数。
- GRPO 用"同 prompt 下的其他 rollouts 平均分"作基线，省一个网络。
- 这是 DeepSeek-R1-Zero 的同款思路。

**为什么 evaluation 用 greedy 不用 sampling？**
- 这个任务有唯一答案，确定性输出最公平。
- Sampling 引入的方差对 1530 题这个规模 → 单个 seed 标准误约 ±1%，3 seed 也只压到 ±0.6%，但成本 ×4。
- 在意 CI 的话可以加 `--with-sampled`，默认关闭。
